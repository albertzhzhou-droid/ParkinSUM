# v0.1.0-alpha Release Notes Draft

ParkinSUM Companion v0.1.0-alpha is planned as a public showcase release for
architecture review, portfolio presentation, educational software engineering
discussion, and synthetic-data demonstration.

## Safety Boundary

This prototype is for educational and research demonstration only. It does not
provide medical diagnosis, treatment recommendation, medication timing advice,
individualized dietary guidance, patient care, emergency support, or personal
health management.

Use only synthetic or sample data. Do not enter real health information,
medication schedules, symptoms, identifiers, Firebase tokens, service account
credentials, user exports, or raw operator audit logs.

## Included In The Alpha

- Flutter app prototype with local-first demo path.
- Onboarding, meal logging, medication context, and timeline-oriented flows.
- Deterministic CDSS-style rule checks for educational conflict awareness.
- Evidence-oriented explanation layer and rule trace architecture.
- Local mode and Firebase-backed internal validation architecture.
- Importer, preflight, release, monitoring, audit, and rollback tooling.
- Public showcase safety documents and contribution rules.

## Build And Run

```sh
git clone https://github.com/albertzhzhou-droid/ParkinSUM.git
cd ParkinSUM
flutter pub get
flutter analyze
flutter test
flutter run -d chrome
```

Optional public preflight:

```sh
npm run public:preflight
```

Optional Firebase rules contract check:

```sh
node tool/firestore_rules_contract_check.mjs
```

## Demo Data Requirement

Public demos must use synthetic/sample data only. Release assets should include
either a sample dataset, a documented demo-mode flow, or screenshots/videos that
show only synthetic values.

## Known Limitations

- No clinical validation is claimed.
- Not reviewed or cleared as a medical device.
- Not intended for real user registration for health use.
- Not intended for real medication, symptom, meal, or patient records.
- Rule and evidence coverage are prototype-scoped and should be reviewed as
  software architecture, not clinical authority.
- APK packaging is planned but not required for local web demo review unless a
  release asset is explicitly attached.

## Suggested GitHub Release Checklist

- Attach APK or clearly state that reviewers should build locally.
- Link README, `DISCLAIMER.md`, `PUBLIC_SHOWCASE_READINESS.md`, and
  `docs/RULE_ENGINE.md`.
- Add screenshots or demo video using synthetic data only.
- Confirm `npm run public:preflight` reports zero `BLOCKER` findings.
- Confirm release notes do not imply diagnosis, treatment, individualized
  clinical advice, or real-world deployment.
